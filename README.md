# Trusted Certificate Authority

## Testing with GCP Confidential Space workload

These steps outline how to test the TCA service locally by getting a real attestation token from a
GCP Confidential Space VM.

### GCP Project Setup

#### Set Project ID and Region

Find your Project ID from the Google Cloud Console (Pantheon). Choose a GCP region (e.g., `us`).

```bash
export PROJECT_ID=YOUR_PROJECT_ID
export YOUR_GCP_REGION=us
```

Replace `YOUR_PROJECT_ID` with your actual Project ID.

#### Install and Initialize Google Cloud SDK (`gcloud`)

Install the Google Cloud SDK if you haven't already:
[https://cloud.google.com/sdk/docs/install](https://cloud.google.com/sdk/docs/install)

Authenticate and configure `gcloud`:

```bash
gcloud auth login
gcloud config set project ${PROJECT_ID}
```

#### Enable Required APIs

```bash
gcloud services enable compute.googleapis.com \
    artifactregistry.googleapis.com \
    confidentialcomputing.googleapis.com \
    --project=${PROJECT_ID}
```

#### Create Artifact Registry Repository

```bash
gcloud artifacts repositories create tca-repo --repository-format=docker \
    --location=${YOUR_GCP_REGION} --description="TCA client images" --project=${PROJECT_ID}
```

#### Configure Docker Authentication

```bash
gcloud auth configure-docker ${YOUR_GCP_REGION}-docker.pkg.dev
```

### Build and Push the Test Client Image

The test client in `tools/tca_gcp_client` will generate a CSR and fetch the attestation token.

```bash
cd tools/tca_gcp_client
docker build -t ${YOUR_GCP_REGION}-docker.pkg.dev/${PROJECT_ID}/tca-repo/tca-client:latest .
docker push ${YOUR_GCP_REGION}-docker.pkg.dev/${PROJECT_ID}/tca-repo/tca-client:latest
cd ../..
```

### Run a Confidential Space VM

Choose a zone within your selected region (e.g., `us-central1-a`).

```bash
export YOUR_ZONE=us-central1-a
```

Create the GCE VM instance. This command enables Confidential Computing and sets the metadata
required to run the test client container.

```bash
gcloud compute instances create cs-tca-client-vm \
    --project=${PROJECT_ID} \
    --zone=${YOUR_ZONE} \
    --machine-type=n2d-standard-2 \
    --confidential-compute \
    --image-project=confidential-space-images \
    --image-family=confidential-space \
    --metadata=tee-image-reference=${YOUR_GCP_REGION}-docker.pkg.dev/${PROJECT_ID}/tca-repo/tca-client:latest,tee-container-log-redirect=true \
    --scopes=cloud-platform \
    --shielded-secure-boot
```

### Get Token and CSR

View the serial console output of the CS VM instance to retrieve the CSR and token generated by the
`entrypoint.sh` script.

```bash
gcloud compute instances get-serial-port-output cs-tca-client-vm \
    --project=${PROJECT_ID} \
    --zone=${YOUR_ZONE} \
    --port=1
```

Look for output similar to:

```text
Base64 CSR:
MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQE...

eyJhbGciOiJSUzI1NiIsImtpZCI6IjJmNDAwN...
Base64 Attestation Token:
```

Create a file named `request.json` with the following content, pasting the values from the console:

```json
{
    "certificateSigningRequest": "<PASTE BASE64 CSR HERE>",
    "attestationEvidence": {
        "gcpAttestationEvidence": {
            "attestationToken": "<PASTE BASE64 ATTESTATION TOKEN HERE>"
        }
    }
}
```

### Clean Up VM Resources

After the VM stops and you have retrieved the token and CSR, the GCE instance resource still exists.
You must delete it to avoid incurring charges.

#### Deleting the VM

```bash
gcloud compute instances delete cs-tca-client-vm \
    --project=${PROJECT_ID} \
    --zone=${YOUR_ZONE} \
    --quiet
```

#### Listing VMs to Confirm

You can list all instances to confirm the deletion. The status should be TERMINATED before it
disappears.

```bash
gcloud compute instances list --project=${PROJECT_ID}
```

### Update TCA Config

An example TCA configuration file can be found at
`javatests/com/google/tca/server/testdata/test_tca_config.json`. It's recommended to copy this file
to the project root as `tca_config.json` and modify it for your needs.

```bash
cp javatests/com/google/tca/server/testdata/test_tca_config.json tca_config.json
```

Get the image digest of the pushed test client image. You can find this in the output of the
`docker push` command or by using the following `gcloud` command:

```bash
gcloud artifacts docker images describe \
    ${YOUR_GCP_REGION}-docker.pkg.dev/${PROJECT_ID}/tca-repo/tca-client:latest \
    --project=${PROJECT_ID}
```

Look for the `image_summary.digest` field in the output (e.g., `sha256:2c30...`).

Add this digest to the `mappings` in your `tca_config.json` file in the project root.

Example entry:

```json
{
    "mappings": {
        "sha256:2c30752e2d1945d3743c97fbb64046551370a1d1e9612560f3fc8b2bab4f80bc": {
            "application_identifier": "test-workload",
            "expiration_time": "2126-01-01T00:00:00Z",
            "application_publisher": "test-publisher"
        }
    }
}
```

### Run TCA Server

#### Run TCA Server on localhost

```bash
bazel run //java/com/google/tca/server:server_main -- local --config-path tca_config.json
```

#### Run TCA Server on AWS

##### TCA Configuration Signing

This section assumes the TCA server is already running in AWS. The AWS infrastructure and server
configuration are managed by Terraform, with definitions located in `operator/aws`.

In the AWS deployment, the TCA service expects its configuration file (`tca_config.json`) to be
signed. The signature, verifiable against the public Sigstore Rekor transparency log, is checked by
the TCA upon loading the configuration. Any updates to the configuration require a new signature to
be generated and deployed.

###### Install the cosign tool

```bash
go install github.com/sigstore/cosign/v2/cmd/cosign@latest
```

Make sure `$HOME/go/bin` is in your `$PATH`.

###### Sign the configuration file

```bash
cosign sign-blob \
  --new-bundle-format=true \
  --rekor-url https://rekor.sigstore.dev \
  --bundle tca_config.json.sigstore.json \
  test_tca_config.json
```

Open the link provided by the `cosign` command in a browser and use OIDC to authenticate.

> **Note:** If the OIDC flow does not complete automatically, for example when accessing the
> terminal via SSH, use the device flow flag:
>
> ```bash
> cosign sign-blob \
>   --new-bundle-format=true \
>   --rekor-url https://rekor.sigstore.dev \
>   --bundle tca_config.json.sigstore.json \
>   --fulcio-auth-flow=device \
>   tca_config.json
> ```

###### Update TCA configuration

To upload the signed configuration and the signature bundle to the AWS S3 bucket used by the TCA,
you can use the following commands. Ensure your AWS CLI is configured with the correct permissions
and region. It is also possible to upload the files via the AWS console

```bash
export AWS_ACCOUNT_ID="${AWS_ACCOUNT_ID:-266438516441}"
export AWS_REGION="${AWS_REGION:-us-east-1}"

BUCKET_NAME="tca-configuration-${AWS_ACCOUNT_ID}-${AWS_REGION}"

aws s3 cp "tca_config.json" "s3://${BUCKET_NAME}/tca_config.json" && \
aws s3 cp "tca_config.json.sigstore.json" "s3://${BUCKET_NAME}/tca_config.json.sigstore.json"
```

### Call TCA with curl

To call the TCA server, you can use the REST API.

#### Calling TCA Server on localhost

```bash
curl -X POST http://localhost:50051/v1/certificates:issue \
  -H "Content-Type: application/json" \
  -d @request.json \
| tee issue_certificate_response.json
```

#### Calling TCA Server on AWS

Authentication via AWS IAM (SigV4) is required. You must assume an authorized client role before
sending requests.

**Assume Role & Export Credentials:**

```bash
# Replace with your client role ARN
ROLE_ARN="arn:aws:iam::030237092795:role/tca-dev-client-dev-admin"

CREDENTIALS=$(aws sts assume-role --role-arn "${ROLE_ARN}" --role-session-name TCASession --output json)
export AWS_ACCESS_KEY_ID=$(echo $CREDENTIALS | jq -r .Credentials.AccessKeyId)
export AWS_SECRET_ACCESS_KEY=$(echo $CREDENTIALS | jq -r .Credentials.SecretAccessKey)
export AWS_SESSION_TOKEN=$(echo $CREDENTIALS | jq -r .Credentials.SessionToken)
```

**Send Authenticated Request:**

```bash
# Replace with your region and domain
REGION="us-east-1"
DOMAIN="tca.dev.pcit.pstest.dev"

curl --aws-sigv4 "aws:amz:${REGION}:execute-api" \
  --user "${AWS_ACCESS_KEY_ID}:${AWS_SECRET_ACCESS_KEY}" \
  -H "x-amz-security-token: ${AWS_SESSION_TOKEN}" \
  -H "Content-Type: application/json" \
  -X POST "https://${DOMAIN}/v1/certificates:issue" \
  -d @request.json \
| tee issue_certificate_response.json
```

#### Inspecting results

First, convert the output to a DER file and then to PEM format:

```bash
cat issue_certificate_response.json | ./tools/issue_certificate_response_to_der.py > cert.der
openssl x509 -inform DER -in cert.der -out cert.pem
openssl x509 -in cert.pem -text -noout
```

To verify the issued certificate against the TCA's root certificate, you can fetch the
`tca_root_certificate.pem` file from S3:

```bash
export AWS_ACCOUNT_ID="${AWS_ACCOUNT_ID:-266438516441}"
export AWS_REGION="${AWS_REGION:-us-east-1}"
aws s3 cp s3://tca-root-key-backup-${AWS_ACCOUNT_ID}-${AWS_REGION}/encrypted/tca_root_certificate.pem tca_root_certificate.pem
```

Then, verify the chain:

```bash
openssl verify -CAfile tca_root_certificate.pem cert.pem
```

This should display the certificate details, including the SPIFFE ID in the Subject Alternative
Name.
